Insurance Policy & Claim Management System — Implementation Plan ASP.NET Core 10

Context

The repo at C:\Tools\CapStoneProject currently holds only the default ASP.NET Core 10 Web API scaffold (WeatherForecast template). The Capstone SRS (Capstone Project.pdf) specifies a full Insurance Policy and Claim Management System: a layered REST API with JWT auth, 3 roles (Admin/Agent/Customer), 9 domain entities, a policy lifecycle, a claim lifecycle, simulated payments, soft-delete, centralized exceptions, structured logging, Swagger, and pagination/sorting/filtering on every list API.

The SRS names Spring Boot, but the user chose to build in ASP.NET Core 10 (mapping Spring concepts to ASP.NET equivalents). This plan delivers the system in the user's requested build order: entities → DTOs → repositories → services → controllers → auth, plus the mandatory cross-cutting concerns.

Tech decisions (confirmed)

  • Persistence: EF Core + SQL Server LocalDB, code-first migrations.
  • API docs: Swashbuckle Swagger UI at /swagger with JWT bearer "Authorize" button.
  • Mapping: AutoMapper profiles.
  • Validation: DataAnnotations on DTOs + automatic ModelState validation.
  • Password hashing: ASP.NET Core PasswordHasher<T> (no extra dependency).

Step 1 — Entities (9 + enums)

Create under Entities/. Use int PKs, DateTime audit fields, navigation properties matching the SRS Entity Relationship Mapping (§10). Soft-delete via IsActive/status fields (DRD rules — never hard delete).

  • User — Id, FullName, Email (unique), PasswordHash, MobileNumber, Role (enum), IsActive, CreatedDate, UpdatedDate. 1:1 Customer.
  • Customer — Id, UserId (FK, unique), DateOfBirth, Address, City, State, PinCode, NomineeName, NomineeRelation, audit. (§9.2)
  • InsuranceProduct — Id, ProductName (unique), ProductType (enum), Description, IsActive, audit. 1:M Plans. (§9.3)
  • PolicyPlan — Id, InsuranceProductId (FK), PlanName, CoverageAmount, PremiumAmount, PremiumType (enum), DurationYears, TermsAndConditions, IsActive, audit. (§9.4)
  • Policy — Id, PolicyNumber (unique), CustomerId (FK), PolicyPlanId (FK), StartDate, EndDate, PolicyStatus (enum), TotalPremiumPaid, audit. (§9.5)
  • PremiumPayment — Id, PolicyId (FK), Amount, PaymentDate, PaymentMode (enum), TransactionReference (unique), PaymentStatus (enum), CreatedDate. (§9.6)
  • Claim — Id, ClaimNumber (unique), PolicyId (FK), ClaimAmount, ClaimReason, IncidentDate, ClaimStatus (enum), AgentRemarks, AdminRemarks, audit. (§9.7)
  • ClaimDocument — Id, ClaimId (FK), DocumentName, DocumentType, DocumentReference (filename/URL/text), UploadedDate. (§9.8)
  • ClaimStatusHistory — Id, ClaimId (FK), PreviousStatus, NewStatus, Remarks, UpdatedByUserId (FK), UpdatedDate. Append-only. (§9.9)

Enums (Entities/Enums/): UserRole {Admin,Agent,Customer}, ProductType {Health,Motor,Life,Travel}, PremiumType {OneTime,Annual}, PolicyStatus {PendingPayment,Active,Expired,Cancelled}, PaymentMode {UPI,Card,NetBanking,Cash}, PaymentStatus {Success,Failed,Pending}, ClaimStatus {Submitted,UnderReview,RecommendedForApproval,RecommendedForRejection,Approved,Rejected}.

Step 2 — DTOs (per §11)

Create under DTOs/<Module>/ with DataAnnotations validation (§17). DTOs must never expose entities or passwords (DTO-RUL-005). Key sets:

  • Common/: ApiResponse<T> (success, message, data, timestamp), ErrorResponse (timestamp, statusCode, errorType, message, path), PagedResponse<T> (records, currentPage, pageSize, totalRecords, totalPages, isLastPage, sortField, sortDirection), PageQuery (pageNumber, pageSize, sortBy, sortDir + module filter fields).
  • Auth: RegisterRequest (no role field — customers only), LoginRequest, LoginResponse (token, tokenType, email, role, expiry).
  • User: UserResponse, CreateAgentRequest, UserStatusUpdateRequest.
  • Customer: CustomerRequest, CustomerResponse.
  • Product: ProductRequest, ProductResponse.
  • Plan: PlanRequest, PlanResponse.
  • Policy: CustomerPolicyPurchaseRequest (plan + start date; customer derived from token), AgentPolicyIssueRequest (+customer ref), PolicyResponse.
  • Payment: PaymentRequest, PaymentResponse.
  • Claim: ClaimRequest, ClaimReviewRequest (agent), ClaimFinalDecisionRequest (admin), ClaimResponse.
  • ClaimHistory: ClaimStatusHistoryResponse.

Validation specifics from §17: email valid/unique, DOB past date, coverage/premium/amount/duration > 0, incident date not future, claim amount ≤ coverage, at least one supporting document.

Step 3 — Data layer + Repositories

  • Data/AppDbContext: DbSet per entity; Fluent API for unique indexes (Email, ProductName, PolicyNumber, ClaimNumber, TransactionReference), relationships (§10 MAP-001..009), Restrict/NoAction delete behavior to enforce no-hard-delete intent.
  • Data/DbInitializer: seed one Admin user (ASM-004 — admin/agent created internally), since public registration is customer-only.
  • Repositories (Repositories/): IGenericRepository<T> (Add/Update/GetById/Query/SaveChanges) + a paged/filtered/sorted query helper (REP-RUL-006). Per-entity interfaces for business-identifier lookups (REP-RUL-007): GetByPolicyNumber, GetByClaimNumber, GetByTransactionReference, ExistsByEmail, etc. No business logic here (REP-RUL-001).
  • First EF migration + database update against LocalDB.

Step 4 — Services (business logic + rules)

Interfaces + impls under Services/. Each enforces its module's business rules (SRV-RUL-001..011), maps via AutoMapper, throws custom exceptions, logs business events, and returns DTOs. Services:

  • AuthService — register (force Customer role), login (reject inactive — USR-BR-005), hash/verify password, issue JWT.
  • UserService — list (paged/filtered by role+active), create agent, activate/deactivate.
  • CustomerService — create/update/view own profile; enforce one-profile-per-user (CUS-BR-002) and owner-only update (CUS-BR-004 / SRV-RUL-010).
  • ProductService — CRUD + deactivate; unique name; only-active-usable (PRD rules).
  • PlanService — CRUD + deactivate; validations PLN-BR-002..004; only under active product.
  • PolicyService — purchase (customer-from-token) / issue (agent/admin); generate unique PolicyNumber; create as PendingPayment (POL-BR-004); compute EndDate from start + plan duration; cancel; expiry check by date (POL-BR-006).
  • PaymentService — record simulated payment; unique transaction ref; on first Success ≥ required premium → activate policy + increase TotalPremiumPaid (PAY-BR-004/007, PMT rules); failed/pending never activate.
  • ClaimService — raise (only own Active policy; amount ≤ coverage; incident not future; ≥1 doc — CLM rules); agent review/recommend; admin final approve/reject; block modification of Approved/Rejected (CLM-BR-009); enforce claim lifecycle transitions (CLC-RUL-001..008).
  • ClaimStatusHistoryService — append a history row on every claim status change (CLC-RUL-007, HIS rules); paged read for admin/agent (own-only for customer).

Pagination/sorting/filtering validated centrally (PAG/SRT rules): default page 1, size 10, max 100, default sort CreatedDate desc; reject invalid page/size/sort field.

Step 5 — Controllers (REST APIs)

Thin controllers per module under Controllers/ (CON-RUL-006: no business logic). Attribute routing [Route("api/[controller]")], [Authorize(Roles=...)] per the §13 access matrix, [ProducesResponseType] for Swagger, return ApiResponse<T> with correct status codes (§18.3). Controllers + capabilities (§22):

  • AuthController — register, login (anonymous).
  • UsersController — Admin: list/create-agent/activate/deactivate.
  • CustomersController — customer profile CRUD-own; admin/agent view/list.
  • ProductsController, PlansController — Admin manage; all roles view active.
  • PoliciesController — purchase (Customer), issue/cancel/list (Admin/Agent), view own (Customer).
  • PaymentsController — record (Customer own / Agent), list & view.
  • ClaimsController — raise (Customer), review+recommend (Agent), final decision (Admin), list/view.
  • ClaimHistoryController — view status history (own-only for Customer).

Step 6 — Auth + cross-cutting (wired in Program.cs)

  • JWT (Security/): JwtTokenService builds tokens with identity, email, role, iat, exp (§23.3 — never include password/sensitive data). appsettings.json Jwt section (Issuer, Audience, Key, ExpiryMinutes). Configure AddAuthentication().AddJwtBearer(...) + AddAuthorization. Reject missing/invalid/expired (SEC-AUTH-004).
  • Current user accessor to derive customer identity from token (ambiguity checklist — customer never passes own ID).
  • GlobalExceptionMiddleware (Exceptions/): map custom exceptions (NotFound→404, Duplicate→409, Validation→400, Unauthorized→401, Forbidden→403, BusinessRule→400/409, fallback→500) to the standard ErrorResponse (§18). Covers EXC-001..019.
  • Logging: built-in ILogger structured logs for LOG-001..017; never log passwords/JWTs (LOG-RUL-001/002).
  • Swagger: AddSwaggerGen with API title/description/version, JWT bearer security definition + "Authorize" button, XML comments, role notes (SWG-001..011); expose in Development.
  • AutoMapper: register mapping profiles from Mapping/.
  • Program.cs middleware order: HTTPS → Swagger (dev) → exception middleware → Authentication → Authorization → MapControllers. Call DbInitializer seed on startup.

Verification

  1. dotnet restore then dotnet build from Backend/InsurancePolicyApi/ — clean build.
  2. dotnet ef migrations add InitialCreate + dotnet ef database update — schema created in LocalDB.
  3. dotnet run --project InsurancePolicyApi → open https://localhost:7083/swagger.
  4. End-to-end happy path via Swagger:
    • Register customer → login → copy JWT → click Authorize.
    • As seeded Admin: create product → create plan.
    • As Customer: create profile → purchase policy (status PendingPayment) → record Success payment ≥ premium → policy flips to Active.
    • Raise claim on active policy → as Agent recommend → as Admin approve → confirm claim status history recorded and approved claim is immutable.
  5. Negative checks: register-as-admin rejected; customer accessing another customer's data → 403/structured error; claim amount > coverage → 400; failed payment does not activate; list APIs return paginated metadata and reject invalid page/sort inputs.
  6. Confirm acceptance criteria AC-001..AC-030 (§30) are observable through the above.

Notes / scope

This is a large multi-module build (~60–80 files). Suggested execution follows the requested order (entities → DTOs → repositories → services → controllers → auth), building cross-cutting (exceptions/logging/swagger) alongside auth. Out-of-scope per SRS: real payment gateway, KYC, notifications, cloud storage, microservices, Docker, renewals, refunds (OOS-001..012).
